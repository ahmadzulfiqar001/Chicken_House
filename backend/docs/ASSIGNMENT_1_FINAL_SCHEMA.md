# Assignment 1 - Final MongoDB Schema and Relationships

## Project Context

This schema write-up is based on the **actual MongoDB Atlas collections** visible in the Chicken House database and the way the current project backend uses or extends them. For assignment purposes, this is the most complete version because it includes:

- all main collections
- their fields and data types
- required and optional fields
- nested entities
- full relationships between collections
- a short explanation of why the structure was designed this way

## Main Collections In The Project

- `categories`
- `ingredients`
- `menuitems`
- `recipes`
- `orders`
- `restaurantsettings`
- `users`

## 1. `categories`

Purpose: stores menu categories such as Pasta, Fried Chicken, BBQ, Pizza, Drinks, etc.

### Fields

- `_id`: ObjectId, required
- `name`: String, required
- `slug`: String, required
- `description`: String, optional
- `imageUrl`: String, optional
- `sortOrder`: Number, optional
- `isActive`: Boolean, optional
- `createdAt`: Date, optional
- `updatedAt`: Date, optional

### Notes

- `slug` is used for cleaner category identification.
- `sortOrder` controls category display order.
- `isActive` allows categories to be hidden without deleting them.

## 2. `ingredients`

Purpose: stores stock and cost information for raw items used in recipes and inventory control.

### Fields

- `_id`: ObjectId, required
- `name`: String, required
- `unit`: String, required
- `currentStock`: Number, required
- `reorderLevel`: Number, required
- `costPerUnit`: Number, required
- `supplierName`: String, optional
- `isActive`: Boolean, optional
- `createdAt`: Date, optional
- `updatedAt`: Date, optional

### Notes

- `currentStock` shows currently available quantity.
- `reorderLevel` helps detect low-stock situations.
- `costPerUnit` is used for cost calculation and profitability.

- `sortOrder`: Number, optional
- `portions`: Array, optional
- `addons`: Array, optional
- `tags`: Array, optional
- `spiceLevel`: Number, optional

- `isFeatured`: Boolean, optional

### Nested Structures

#### `portions[]`

- `label`: String, required

Example meaning:
- Half
- Full
- 2 Person
- 4 Person

#### `addons[]`

- `name`: String, required

Example meaning:

## 4. `recipes`

Purpose: connects menu items to ingredients and defines how much stock is consumed when an item is prepared.

### Fields

- `_id`: ObjectId, required
- `menuItem`: ObjectId, required
- `ingredients`: Array, required
- `wastePercent`: Number, optional
- `deductionEnabled`: Boolean, optional
- `createdAt`: Date, optional
- `updatedAt`: Date, optional

### Nested Structures

#### `ingredients[]`

- `ingredient`: ObjectId, required
- `quantity`: Number, required
- `unit`: String, optional

### Notes

- `menuItem` connects the recipe to one menu item.
- `ingredients[]` contains multiple ingredient references.
- `wastePercent` is used to account for cooking/prep waste.
- `deductionEnabled` determines whether stock should actually be reduced on order placement.

## Full Relationships Between Collections

## A. Core Food Hierarchy

- `categories._id` -> `menuitems.category`
  Meaning: each menu item belongs to one category.

- `menuitems._id` -> `recipes.menuItem`
  Meaning: each recipe belongs to one menu item.

- `ingredients._id` -> `recipes.ingredients[].ingredient`
  Meaning: each recipe contains one or more ingredient references.

## B. Order Flow Relationships

- `menuitems` -> `orders.details[]`
  Meaning: ordered products come from menu items.

- `orders.details[].menuItemId` -> `menuitems._id`
  Meaning: if stored, this is the direct item-level reference from order detail to menu item.

- `recipes` + `orders.details[]` -> inventory deduction logic
  Meaning: when an order is placed, the ordered menu item is matched with its recipe, and the recipe ingredients are deducted from `ingredients.currentStock`.

## C. Business Configuration Relationships

- `restaurantsettings` -> website branding
  Meaning: business name, logo, tagline, colors, contact info, and hero content come from this collection.

- `restaurantsettings.openingHours[]` -> website/service timing display
  Meaning: opening hours are shown on site and used for customer information.

- `restaurantsettings.socialLinks[]` -> website footer/contact links
  Meaning: social accounts are displayed publicly.

- `restaurantsettings.branches[]` -> branch/location information
  Meaning: branch data is stored inside settings unless separated later.

## D. User/Admin Relationships

- `users` -> admin panel authentication
  Meaning: users collection stores login and role information.

- `users.role` -> access control logic
  Meaning: permissions such as create/update/delete depend on user role.

## E. Logical End-to-End Chain

The most important relationship chain in the system is:

`categories` -> `menuitems` -> `recipes` -> `ingredients` -> `orders`

Detailed meaning:

1. Categories organize the menu.
2. Menu items define what customers can order.
3. Recipes define what ingredients are needed for each menu item.
4. Ingredients store current stock and cost.
5. Orders consume menu items, which indirectly consume ingredients through recipes.

## Relationship Summary Table

| Parent Entity | Child Entity | Connected By | Relationship Type |
| --- | --- | --- | --- |
| `categories` | `menuitems` | `menuitems.category` | One-to-Many |
| `menuitems` | `recipes` | `recipes.menuItem` | One-to-One or One-to-Many |
| `ingredients` | `recipes.ingredients[]` | `ingredients[].ingredient` | Many-to-Many through recipe list |
| `menuitems` | `orders.details[]` | `menuItemId` or logical item match | One-to-Many |
| `restaurantsettings` | `socialLinks[]` | embedded array | One-to-Many |
| `restaurantsettings` | `openingHours[]` | embedded array | One-to-Many |
| `restaurantsettings` | `branches[]` | embedded array | One-to-Many |
| `users` | admin/auth actions | login and role usage | One-to-Many logical usage |

## Required vs Optional Design Pattern

The schema follows a practical design rule:

- fields needed to identify or connect records are required
  Example: `_id`, `menuitems.category`, `recipes.menuItem`, `recipes.ingredients[].ingredient`

- fields that improve display, filtering, or business usability are optional
  Example: `description`, `imageUrl`, `sortOrder`, `supplierName`, `notes`

- arrays and embedded objects are used where child data naturally belongs to one parent
  Example: `openingHours[]`, `socialLinks[]`, `branches[]`, `portions[]`, `addons[]`

## Why This Structure Was Chosen

This database structure fits the Chicken House project because it separates the restaurant system into clear modules. Categories and menu items control what is shown to customers, while recipes and ingredients support kitchen operations and stock deduction. Orders are stored separately because they are transactional records that grow over time and need status tracking. Restaurant settings are kept in their own collection because they store global business configuration, and users are isolated for authentication and role management.
