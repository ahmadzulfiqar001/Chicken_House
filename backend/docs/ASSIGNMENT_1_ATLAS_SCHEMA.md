# Assignment 1 - Actual Mongo Atlas Schema

## Why This File Exists

This file documents the **actual MongoDB Atlas collections currently present** in the Chicken House database shown in Atlas. It is the best match for the assignment because it describes the real project database, not only the normalized compatibility models used by the current backend.

## Relationships Between Actual Atlas Collections

- `categories._id` -> `menuitems.category`
- `menuitems._id` -> `recipes.menuItem`
- `ingredients._id` -> `recipes.ingredients[].ingredient`
- `menuitems` -> `orders.details[]`
  Note: this relationship is implemented by the app logic even though the Atlas screenshot does not show order documents yet.
- `restaurantsettings` is a global configuration collection and stands mostly standalone.
- `users` is the authentication/admin collection.

## Why The Database Is Structured This Way

The Atlas database is structured around the main restaurant workflow. Categories organize menu items, menu items define the dishes customers can order, ingredients track stock, and recipes connect each dish to the ingredients it consumes so inventory can be managed properly. Restaurant settings are kept separately because they represent global business configuration rather than transactional data. Users are also separated because authentication and admin access should stay independent from menu and stock records.
